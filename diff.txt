+haskell webapi Clean Architecture 
+
+方向性を整理しつつ、2026年時点でも通用する Haskell の Web API 構成を確認しています。主要ライブラリの現状を当たり、Clean Architecture に落とし込んだ実践形でまとめます。
+
+主要候補は見えました。型で API 契約を先に置ける Servant が Clean Architecture と相性がよく、Scotty は薄いルーティング層として軽量ですが、境界を自分で厳密に保つ必要が強めです。
+
+思考時間: 24s
+
+Haskell で Web API を Clean Architecture で組むなら、まず結論はこれです。
+
+いちばん実践的なのは Servant + WAI/Warp + effectful + PostgreSQL層 です。
+Servant は API を型レベルで宣言でき、サーバ実装・クライアント生成・ドキュメント生成まで一貫して扱えるので、「外側の入出力」と「内側のユースケース」を分離する Clean Architecture とかなり相性がいいです。Servant 自体も、型安全に API 定義・サーバ・クライアント・ドキュメントを扱うことを主眼にしています。
+
+一方で Scotty は 軽量で始めやすい ですが、README でも WAI/Warp ベースのシンプルな Web フレームワークとして位置づけられており、API 契約や境界の厳密さは Servant ほど自動では得にくいです。小規模ならよいですが、Clean Architecture を強く維持したいなら Servant のほうが素直です。

+ポイントは、Servant の API 型を Domain に置かないことです。
+API 型はあくまで外側の表現なので Interface/Api.hs に置くのが無難です。
+
+役割分担の具体像

+Servant を選ぶ理由
+
+Servant は API 契約を型として先に置けるので、Clean Architecture でいう「外部との境界」が明確になります。
+さらに Servant は、同じ API 定義からサーバ実装やクライアント生成、ドキュメント生成に展開できる設計思想を持っています。これが「境界の一元管理」と相性がいいです。
+
+感覚的にはこうです。
+
+Scotty: ルートを書きながらアプリを作る
+
+Servant: 契約を書いてから実装を埋める
+
+Clean Architecture は後者の流れに乗せやすいです。
+
+effectful を使う理由
+
+Haskell では ReaderT Env IO も定番ですが、規模が大きくなると
+
+DB
+
+Logger
+
+Clock
+
+Auth
+
+Transaction
+
+External API
+
+のような関心が増えます。
+
+effectful は高速性と既存ライブラリとの統合を前面に出していて、ユースケースから必要な能力だけ要求する形にしやすいです。
+そのため、
+
+Domain は純粋
+
+UseCase は抽象的な効果
+
+Infrastructure で実行
+
+という分離がしやすいです。
+
+最小の設計パターン
+
+実際には次の 3 つだけ守るとかなり崩れません。
+
+1. Domain は純粋にする
+
+aeson
+
+servant
+
+opaleye
+
+persistent
+
+を Domain まで入れない
+
+2. UseCase は HTTP を知らない
+
+たとえば 404 や 401 を直接返さず、AppError を返す。
+
+data AppError
+  = UserNotFound
+  | InvalidEmail
+  | Unauthorized
+
+HTTP への変換は Interface 側。
+
+3. Infrastructure は抽象を実装するだけ
+
+DB 実装や JWT 実装は UseCase が定義したポートに従わせる。
+
+典型的なリクエストの流れ
+HTTP Request
+  -> Servant Handler
+  -> DTO decode
+  -> UseCase 呼び出し
+  -> Repository / Auth / Clock などのポート使用
+  -> Domain result / AppError
+  -> Presenter で HTTP Response 化
+
+これで「外の表現」と「中のルール」が混ざりにくくなります。
+
+ありがちな失敗
+1. Domain に JSON/DB インスタンスを全部載せる
+
+最初は楽ですが、あとで層が溶けます。
+
+2. Servant Handler の中で SQL を書く
+
+これは Clean Architecture 的にはかなり崩れやすいです。
+
+3. Env をどこでも触れる万能箱にする
+
+依存が見えなくなります。
+UseCase には必要な能力だけ渡すほうが保守しやすいです。
+
+4. DTO と Domain を同一化する
+
+外部公開 API の都合で Domain が歪みます。
+
+2026年時点での現実的な選択
+
+私ならこうします。
+
+堅めの本命
+
+servant-server
+
+wai
+
+warp
+
+effectful
+
+servant-auth-server
+
+PostgreSQL
+
+opaleye
+
+軽量に始める
+
+scotty
+
+wai
+
+warp
+
+ReaderT Env IO
+
+ただし後者は、長く育てる API だと境界管理を自分でかなり厳格にやる必要があります。
+Scotty は軽量さが魅力ですが、Servant のような API 契約中心の恩恵は薄いです。
+
+まず作るならこの順番
+
+Domain にエンティティとエラーを書く
+
+UseCase にポートを書く
+
+UseCase に業務ロジックを書く
+
+Interface に Servant API と DTO を書く
+
+Infrastructure に DB/JWT 実装を書く
+
+Main で全部組み立てる
+
+この順序にすると、Web フレームワークに引っ張られずに済みます。
+
+ひとことで言うと
+
+Haskell で Clean Architecture をやるなら、Servant を「外側の契約」、UseCase を「中心」、effectful を「依存の接着剤」として使うのがかなり強いです。
+Servant は API 契約の型安全さとドキュメント・クライアント生成の一貫性が強みで、effectful はユースケースの依存を整理しやすく、認証は servant-auth 系、DB は Postgres 向けに Opaleye のような型安全 DSL が有力です。
+
+必要なら次に、**「Servant + effectful で作る最小 Web API の雛形」**をそのまま cabal/stack で動く形で出します。
